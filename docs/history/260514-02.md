## Q13. 현재 아키텍처 이해 확인 — 로컬 vs 외부 저장소

**질의:** `내가 궁금한 건, 지금은 마치 skills 문서가 내쪽에 있고, 외부 저장소에 해당 지식이 저장되어 있고, 내 로컬에서 외부를 찔러서 가져오는 구조로 가야 되는걸로 이해하고 있어. 이 그림이 맞는거야?`

**답변:**

반은 맞고, 반은 틀리다.

**ocpp.md 방식의 실제 구조 (외부 아님, 전부 로컬):**
```
로컬 (Claude Code 실행 환경)
  ├── skills/ocpp/SKILL.md     ← 슬래시 커맨드 정의
  └── docs/ocpp/               ← 문서도 로컬에 있음
        ├── OCPP-1.6J-Core.md
        └── OCPP-2.0.1-SmartCharging.md
```

`${CLAUDE_PLUGIN_ROOT}/docs/xxx.md`는 외부 서버 API 호출이 아니라, **로컬 파일시스템에서 Read 도구로 파일을 읽는 것**이다.

**두 아키텍처 비교:**

| | ocpp.md 방식 (현재) | RAG 방식 (3단계 학습 목표) |
|---|---|---|
| 문서 위치 | 로컬 .md 파일 | 외부 벡터 DB (ChromaDB 등) |
| 검색 방법 | Claude가 직접 Read/Grep | 임베딩 → 벡터 유사도 검색 |
| 구현 난이도 | 낮음 | 높음 |
| 문서 규모 | 수십 개 파일 수준 | 수천~수만 청크 가능 |

```
현재 이해 단계:  로컬 .md → Claude가 Read → 답변
다음 단계 (RAG): 질문 → 벡터 DB 검색 → 관련 청크 → 답변
```

## Q14. 바로 RAG 학습하는 게 더 좋을까?

**질의:** `시간 단축 차원에서 바로 RAG 학습하는게 더 좋을거 같은데?`

**답변:**

추천은 **로컬 파일 Read 방식으로 먼저 동작하는 것 하나 만들고 나서 RAG로 가는 것**.

RAG는 "검색 → 답변" 파이프라인 전체를 한 번에 구축해야 해서, 슬래시 커맨드 자체가 처음이면 어디서 막혔는지 디버깅이 매우 어렵다.

| | 로컬 파일 먼저 | 바로 RAG |
|---|---|---|
| 시간 | 1~2일이면 동작하는 것 완성 | 벡터 DB 셋업만 2~3일 |
| 실패 지점 | 명확 | 여러 레이어가 겹쳐서 불명확 |
| 결과 | OCPP 스펙 30~40개 문서면 충분히 커버 | 문서가 수천 페이지 이상일 때 진가 발휘 |

OCPP 스펙 문서는 분량이 크지 않아서 로컬 파일 Read 방식으로도 충분히 실용적. RAG가 필수인 상황이 아님.
