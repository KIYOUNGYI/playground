## Q1. RAG 백엔드 동작 원리

**질의:** `RAG 를 구축하고 거기에 ocpp 도메인 지식을 넣었다 쳐. 만약 사용자가 claude 에 질의했다면, 백엔드에서는 어떤 원리로 질의해서 사용자한테 결과값을 리턴해?`

**답변:**

사용자가 질의하면 백엔드는 크게 두 단계로 처리한다.

**1단계 — Retrieval (관련 문서 검색)**

```
사용자 질문
    ↓
[Embedding Model]  ← 질문을 벡터로 변환
    ↓
[Vector DB 검색]   ← 코사인 유사도로 가장 가까운 문서 top-k 개 추출
    ↓
관련 OCPP 문서 청크들 (예: "OCPP 1.6 BootNotification 스펙", "Heartbeat interval 설명" 등)
```

- 질문 텍스트를 동일한 Embedding Model로 벡터화 (데이터 적재 때 쓴 것과 같은 모델이어야 함)
- Vector DB (Pinecone, Weaviate, pgvector 등)에서 유사도 높은 청크 k개 반환

**2단계 — Generation (Claude에게 컨텍스트 주입 후 답변 생성)**

```python
retrieved_chunks = vector_db.search(embed(user_query), top_k=5)

prompt = f"""
당신은 OCPP 전문가입니다.
아래 참고 문서를 바탕으로 질문에 답하세요.

[참고 문서]
{'\n\n'.join(retrieved_chunks)}

[질문]
{user_query}
"""

response = claude.messages.create(
    model="claude-sonnet-4-6",
    messages=[{"role": "user", "content": prompt}]
)
```

Claude는 RAG를 모름 — 그냥 "참고 문서 + 질문"이 담긴 프롬프트를 받아서 답할 뿐이다.

**전체 흐름:**

```
사용자 질문
    ↓
① 질문 → 벡터 변환 (Embedding)
    ↓
② Vector DB에서 유사 OCPP 문서 청크 검색
    ↓
③ [참고 문서 + 원본 질문] 조합해서 프롬프트 구성
    ↓
④ Claude API 호출
    ↓
⑤ Claude 응답 → 사용자에게 반환
```

| 항목            | 설명                                |
|---------------|-----------------------------------|
| Claude의 역할    | 검색된 문서를 이해하고 답변 생성 (직접 DB를 보지 않음) |
| Embedding 일관성 | 적재할 때 / 검색할 때 반드시 같은 모델 사용        |
| top-k 튜닝      | 너무 많으면 context 오염, 너무 적으면 정보 부족   |
| 청크 크기         | OCPP 스펙처럼 표/코드가 많으면 청크 경계 설계가 중요  |

OCPP 도메인이면 메시지 타입별 청킹 (BootNotification, RemoteStartTransaction 등 action 단위)이 검색 품질에 유리하다.

## Q2. RAG가 정확히 뭐야?

**질의:** `요즘 RAG 란 단어를 심심찮게 들어. 이게 정확히 뭘 하는거야? 나는 이걸 구축하고 여기에 질의해서 응답값을 내려주는 걸로 이해했었거든.`

**답변:**

**문제: LLM은 지식이 고정되어 있다**

```
Claude 학습 완료 시점: 2025년 8월
→ 그 이후 정보는 모름
→ 우리 회사 내부 문서도 모름
→ OCPP 최신 에라타도 모름
```

그렇다고 Claude를 우리 데이터로 재학습(fine-tuning)시키는 건 비용이 수억 원이고, 데이터 바뀔 때마다 다시 해야 한다.

**RAG가 해결하는 것**

> "학습 안 시켜도 되게, 답변할 때 관련 문서를 그때그때 꺼내서 읽혀주자"

```
기존 방식:  사용자 질문 → LLM → 답변  (LLM이 아는 것만 답함)

RAG 방식:   사용자 질문 → 문서 검색 → [문서 + 질문] → LLM → 답변
                              ↑
                         Vector DB (우리가 쌓은 지식 저장소)
```

LLM 자체는 바꾸지 않는다. 프롬프트에 관련 문서를 끼워 넣는 것이 전부다.

**Vector DB가 핵심 — 의미 기반 검색**

```
일반 DB:   "BootNotification" 키워드가 있는 문서 검색  (exact match)

Vector DB: "충전기가 서버에 처음 연결될 때 뭘 보내?"
           → 의미상 가장 가까운 문서 검색
           → "BootNotification은 충전기 부팅 시 전송하는 메시지..." 반환
```

텍스트를 숫자 벡터로 바꿔서 저장해두고, 질문도 벡터로 바꿔서 거리 계산으로 가장 가까운 것을 찾는다.

**네 이해에서 빠진 부분**

| 네 이해               | 실제                                   |
|--------------------|--------------------------------------|
| RAG에 질의하면 RAG가 답한다 | RAG는 검색만 한다. 답변은 여전히 LLM(Claude)이 한다 |
| 지식 저장소 = 새로운 AI    | 지식 저장소 = 그냥 검색 엔진 (의미 기반)            |
| RAG가 학습한다          | 학습 없음. 문서를 벡터로 변환해서 저장할 뿐            |

**한 줄 요약:** RAG = "답변 전에 관련 문서를 찾아서 LLM한테 참고하라고 건네주는 파이프라인"

## Q3. LLM에 보내기 전 질문을 '가공'하는 게 핵심?

**질의:** `ocpp.md 스킬처럼 / 사용자가 질문 -> <문서 검색(벡터 디비) -> (문서 + 질문)> // LLM 에 질문하기 전까지 질문을 '가공' 하는게 핵심인거네?`

**답변:**

정확하다. 네가 이미 ocpp.md 스킬로 RAG의 개념을 구현하고 있었다.

```
ocpp.md 스킬:  사용자 질문 → [관련 .md 파일 Read] → [문서 + 질문] → Claude
RAG:           사용자 질문 → [Vector DB 검색]      → [청크 + 질문] → Claude
```

구조가 동일하고, 차이는 검색 방법뿐이다.

|       | ocpp.md 스킬         | RAG              |
|-------|--------------------|------------------|
| 검색 주체 | Claude가 직접 파일 Read | Vector DB가 자동 검색 |
| 검색 기준 | 파일명/구조를 Claude가 판단 | 의미 유사도 (자동)      |
| 문서 규모 | 수십 개 파일            | 수만 개 청크도 가능      |
| 가공 결과 | 프롬프트에 문서 주입        | 동일               |

"질문 가공"을 업계에서는 **컨텍스트 주입**이라고 부른다. RAG든 ocpp.md 스킬이든 그 "붙이는 과정"을 자동화/확장하는 방법의 차이일 뿐, 본질은 같다.
