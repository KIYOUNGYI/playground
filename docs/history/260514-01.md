## Q5. Q2의 1번 항목 설명

**질의:** `@docs/history/260514-01.md 의 Q2 에서 1번 내용에 대해 설명해줘.`

**답변:**

Q2의 1번 항목은 **Claude Code 슬래시 커맨드**입니다.

### Claude Code 슬래시 커맨드란?

Claude Code CLI에서 `/명령어` 형태로 실행하는 커스텀 명령어입니다.

예시:
```
/ocpp1.6 StartTransaction 스펙 알려줘
```

이 프로젝트에서는 `/ocpp1.6` 같은 커맨드를 만들어, OCPP 관련 질문을 AI가 내부 문서 기반으로 답변하도록 하는 것이 목표입니다.

### 어떻게 만드나요?

슬래시 커맨드는 `.claude/commands/` 디렉토리에 마크다운 파일로 정의합니다.

```
.claude/
  commands/
    ocpp1.6.md   ← /ocpp1.6 커맨드 정의
```

`ocpp1.6.md` 파일 안에 프롬프트 내용을 작성하면, `/ocpp1.6` 호출 시 그 프롬프트가 실행됩니다.

```markdown
OCPP 1.6 전문가로서 다음 질문에 답해주세요: $ARGUMENTS
```

`$ARGUMENTS`는 커맨드 뒤에 입력한 텍스트가 자동으로 들어가는 변수입니다.

### CLAUDE.md와의 관계

`CLAUDE.md`는 프로젝트 루트에 두는 설정 파일로, Claude Code가 해당 프로젝트에서 어떻게 동작할지 지시합니다. 슬래시 커맨드의 동작 방식이나 규칙을 여기에 정의할 수 있습니다.

**요약:** 1번을 먼저 배우는 이유는, 이 프로젝트의 핵심 진입점(사용자 인터페이스)이 슬래시 커맨드이기 때문입니다. 나머지 RAG, Java AI 등은 이 커맨드 안에서 호출되는 백엔드 로직입니다.

## Q8. ocpp.md 프로젝트의 슬래시 커맨드 처리 방식

**질의:** `@sample_projects/ocpp.md 이 프로젝트에서는 슬래시 커맨드 관련해서 어떻게 처리한거야?`

**답변:**

Q5에서 설명한 단순 `.claude/commands/` 방식과 다른 **플러그인 + Skill** 방식을 사용.

**디렉토리 구조:**
```
.claude-plugin/
  plugin.json       ← 플러그인 메타데이터
  marketplace.json  ← 마켓플레이스 등록 정보

skills/
  ocpp/
    SKILL.md        ← 실제 슬래시 커맨드 정의
```

**설치 방법:**
```
/plugin marketplace add https://github.com/alexeimoisseev/ocpp.md
/plugin install ocpp@ocpp
```

**SKILL.md 핵심 특징:**
- `user-invocable: true` — `/ocpp` 슬래시 커맨드 활성화
- `allowed-tools: Read, Grep, Glob` — 허용 도구 명시 제한
- 버전 자동 감지 — 코드에서 `StartTransaction` → 1.6J, `TransactionEvent` → 2.0.1
- 서브커맨드 라우팅 — `/ocpp 1.6 smart-charging` → 특정 docs 파일 동적 로드
- 에스컬레이션 모델 — 스펙 미정의 영역은 `SPEC-SILENT` 표시 후 개발자에게 질문

**Q5 방식 vs ocpp.md 방식 비교:**

| 항목 | Q5 단순 방식 | ocpp.md 방식 |
|------|-------------|-------------|
| 파일 위치 | `.claude/commands/*.md` | `skills/*/SKILL.md` |
| 배포 | 로컬 전용 | 마켓플레이스 공유 가능 |
| 문서 처리 | 프롬프트에 직접 작성 | `Read` 도구로 동적 로드 |
| 서브커맨드 | 없음 | `/ocpp 1.6`, `/ocpp smart-charging` 등 라우팅 |
| 허용 도구 | 제한 없음 | 명시적 제한 |

→ 우리 프로젝트 목표(내부 문서 기반 응답)에는 ocpp.md 방식이 더 적합한 레퍼런스.

### 1. 프론트매터 (메타데이터)

```yaml
---
name: ocpp
```

### 2. 본문 구성 (= Claude에게 전달되는 시스템 프롬프트)

| 서브커맨드 라우팅 | `/ocpp 1.6 smart-charging` → 해당 docs 파일 Read |
| 에스컬레이션 모델 | SPEC-SILENT / VENDOR-DEPENDENT / POLICY-DEPENDENT 분류 후 질문 |

### 핵심 설계 원칙

문서를 통째로 컨텍스트에 올리지 않고, **필요한 파일만 그때그때 동적으로 Read** → 컨텍스트 낭비 없이 수십 개 스키마 문서 처리 가능.

```
/ocpp 1.6 StartTransaction → 버전 감지(1.6J) → 파일맵 참조
  → Read(OCPP-1.6J-Schemas-Core.md) → 스펙 인용 답변
  → ESCALATE 마커 있으면 개발자에게 질문
```

## Q10. CLAUDE_PLUGIN_ROOT 란?

**질의:** `SKILL.md 에 적힌 CLAUDE_PLUGIN_ROOT 는 무엇을 뜻하는거니?`

**답변:**

플러그인이 설치된 루트 디렉토리의 절대 경로를 가리키는 런타임 변수.

사용자마다 설치 경로가 다를 수 있어서 SKILL.md에 절대경로를 하드코딩하면 다른 환경에서 깨짐.

```markdown
# 하드코딩 (나쁜 예)
Read /Users/alexei/.claude/plugins/ocpp/docs/xxx.md

# 변수 사용 (올바른 예)
Read ${CLAUDE_PLUGIN_ROOT}/docs/xxx.md
```

런타임 동작:
```
${CLAUDE_PLUGIN_ROOT} → 실제 설치 경로로 치환
→ Read /Users/yikiyoung/.claude/plugins/ocpp/docs/OCPP-2.0.1-SmartCharging.md
```

한 마디로: "이 플러그인이 설치된 폴더가 어디든, 거기서부터 상대 경로로 파일을 찾아라."

## Q12. Git Mirror 뜻

**질의:** `mirror 뜻이 뭐니?`

**답변:**

원본 레포의 모든 것(코드, 브랜치, 태그)을 그대로 복사해서 내부에 독립적으로 보관하는 것.

| 항목 | 일반 Clone | Mirror |
|------|-----------|--------|
| 목적 | 개발용 | 백업 / 내부 보관용 |
| 원본과의 관계 | 원본 추적 | 독립적 보관 |

```bash
git clone --mirror https://github.com/alexeimoisseev/ocpp.md
git push --mirror https://gitlab.내부주소/cpos/ocpp.md.git
```

효과: 개발자 PC → (사내망) → 사내 GitLab. 외부 인터넷 직접 참조 없음.

주의: 원본이 업데이트되면 사내 mirror도 수동 동기화 필요. GitLab "Pull Mirror" 기능으로 자동화 가능.
